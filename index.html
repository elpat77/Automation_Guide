Website Overview

This website was built as a place to keep information on how to to run automation using different programing languages.

It contains things such as installation, tools needed and scripts to use in order to run tests.

Important Steps to take before writting automation:

    Number one, it's all about deciding what features to automate. This requires having knowledge of the application under test and all of the available functionality.
    Number two, outline the scenarios that will be automated, and the steps of each test. Be aware of what the inputs and outputs will be, and the value each scenario implemented will provide.
    Number three, find the web elements that will be needed for testing. It's important to figure out what elements are going to be used in the tests upfront. This will make test writing much easier, faster, and smoother

    Using WebDriver

    The Webdriver API docs can be found on the following site:

    Webdriver API Docs

    We will be using the Ruby API documentation for WebDriver. Familiarize yourself with all the classes and methods available to perform actions on the browser.


    WebDriver is the perfect option for those who have a need to test their application across multiple browsers and platforms. There is a lot of customization to use the WebDriver API, which makes it a very powerful testing framework. If you need to automate web tests, WebDriver is definitely the way to go.
    Selenium WebDriver works using client server communication. When a Selenium test is executed, a new session of the browser driver is created and a browser is launched.
    For each command in the test script, a request is sent to the WebDriver API. The WebDriver API interprets the request and the step is then executed in the browser driver, which acts as the server and just waits for the request to come in. Once each step is complete, the response is sent back to the WebDriver API and then back to the test. This process continues until all steps are complete.

    Appium Overview

    Appium is an open-source tool for automating native, mobile web, and hybrid applications on iOS mobile, Android mobile, and Windows desktop platforms. It's actually a collection of a number of different components. The most basic and important distinction to make in the world of Appium is between the Appium Server and the Appium Clients.
    There is just one Appium Server whereas there are a number of different Clients.

    Appium Website

    Appium Concepts

    Client/Server Architecture

    This Client-Server architecture is the secret behind much of Appium's awesomeness.
    Appium is at its heart a webserver that exposes a REST API. It receives connections from a client, listens for commands, executes those commands on a mobile device, and responds with an HTTP response representing the result of the command execution. The fact that we have a client/server architecture opens up a lot of possibilities: we can write our test code in any language that has a http client API, but it is easier to use one of the Appium client libraries. We can put the server on a different machine than our tests are running on. We can write test code and rely on a cloud service like Sauce Labs to receive and interpret the commands.


    Drivers

    Apart from the fact that Appium has a client-server architecture, the most important thing to know about how Appium is designed is that it is built around a number of platform-specific drivers. A driver is a code module that translates the WebDriver protocol into actual automation behaviors for a particular platform. Through the inclusion of a variety of different drivers, Appium supports automation across a wide variety of platforms, though of course, the most popular platforms are iOS and Android.
    The way it works is that your test script makes calls to a single Appium server. Based on the type of platform you want to automate, for example, iOS, Appium will choose an appropriate driver to handle commands for your test


    Session

    Automation is always performed in the context of a session. Clients initiate a session with a server in ways specific to each library, but they all end up sending a POST /session request to the server, with a JSON object called the 'desired capabilities' object. At this point the server will start up the automation session and respond with a session ID which is used for sending further commands.


    Desired Capabilities

    Desired capabilities are a set of keys and values (i.e., a map or hash) sent to the Appium server to tell the server what kind of automation session we're interested in starting up. There are also various capabilities which can modify the behavior of the server during automation. For example, we might set the platformName capability to iOS to tell Appium that we want an iOS session, rather than an Android or Windows one.

  • See the capabilities doc for a complete list:

    Appium Capabilities


    • Appium Clients

    There are client libraries (in Java, Ruby, Python, PHP, JavaScript, and C#) which support Appium's extensions to the WebDriver protocol. When using Appium, you want to use these client libraries instead of your regular WebDriver client.

    Test Set-up

    As already mentioned, Appium has a client-server architecture which means every Appium command gets sent over the network as an http request. We want our test to proceed in a logical manner, one step after the other, finding elements and interacting with them. What this means is that our Appium client code needs to attach something to every request to let the Appium server know we are dealing with the same test. In Appium web driver lingo, that little something is called a session ID and the Appium server will associate any commands sent in with that ID as belonging to the same session.

    We can start many different kinds of Appium sessions, for example on an IOS simulator, or on an Nexus S Android device physically plugged into our computer. So to make sure we're automating the right kind of thing for our test, we need to tell Appium what it is we want. We do this with a form of session parameters known as capabilities.

    Capabilities are essentially a set of keys and values that get wrapped up into an object and sent to the Appium server during session initialization. There are many, many capabilities that Appium supports, well over a hundred. Some capabilities are generic and take effect in any platform Appium supports. However, there's a minimal set of capabilities which is required to start an Appium session on any native mobile app, and those should always be included. These capabilities are:

    • platformName - the value here should be a string noting the mobile platform you want to test on. Usually, it's either IOS or Android.
      capabilities.setCapability("platformName", "Android");
    • platformVersion - a string denoting the version of that mobile operating system that you want to automate. For example, if you're working with IOS it could be 12.1 or 12.2
      capabilities.setCapability("platformVersion", "12.1");
    • deviceName - this is a string that specifies which device Appium should connect to. It could be for example, the Androiud Emulator simulator
      capabilities.setCapability("deviceName", "Android Emulator");
    • automationName - indicate which driver to use, it could be Apple's XCUITest or UiAutomator2Driver for Android
      capabilities.setCapability("automationName", "UiAutomator2");
    • app - a path to the prebuilt binary version of the app that you want to test. It should either be an absolute local path, like /users/foo/app.apk. It can also be a fully qualified URL to a downloadable zipped version of the app
      capabilities.setCapability("app", "/path/to/my.apk");

    After defining our capabilities, we need to actually use them to start an automation session. To start a session, we merely instantiate a class called Appium driver.
    There are two important classes that inherit from Appium driver. Base on which class we use, the java client will make sure that the appropriate platform-specific methods are available to us:

    • 1) Android Driver - In this example, we're starting an Android session and we therefore use the Android driver class.
      AndroidDriver driver = new AndroidDriver (server,capabilities);
    • 2) iOS Driver - In this example, we're starting an iOS session and we therefore use the iOS driver class.
      IosDriver driver = new IosDriver (server,capabilities);

    To start the session, we also need to tell the Appium client where the Appium server is. In our case, we're assuming Appium is running locally, on its default port of 4723.

    • URL server = new URL ("http://localhost:4723/wd/hub");
